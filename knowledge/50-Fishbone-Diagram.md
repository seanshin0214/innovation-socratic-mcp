# Fishbone Diagram (특성요인도 / 이시카와 다이어그램)

## 개요
Kaoru Ishikawa가 개발한 근본 원인 분석 도구로, 문제(결과)와 그 잠재적 원인들을 시각적으로 표현합니다. 물고기 뼈 모양으로 인해 "Fishbone Diagram"이라 불립니다.

## 창시자
- **Kaoru Ishikawa** (石川馨, 1968)
- 일본 품질관리의 아버지
- 저서: "Guide to Quality Control"

## 기본 구조

```
                    ┌─ People (인력)
                    │
                    ├─ Process (프로세스)
                    │
[문제/결과] ◄───────┼─ Technology (기술)
                    │
                    ├─ Culture (문화)
                    │
                    └─ Resources (자원)
```

## 6M 분류 체계 (제조업)

```
      Man          Machine        Material
       │              │              │
       ▼              ▼              ▼
    ────┬─────────────┬─────────────┬────
        │             │             │
        └─────────────┼─────────────┘
                      │
              [품질 문제] ◄──────────────
                      │
        ┌─────────────┼─────────────┐
        │             │             │
    ────┴─────────────┴─────────────┴────
       ▲              ▲              ▲
       │              │              │
    Method      Measurement    Mother Nature
                               (Environment)
```

| 범주 | 설명 | 검토 질문 |
|------|------|----------|
| Man (인력) | 기술, 훈련, 동기부여 | 담당자의 역량은 충분한가? |
| Machine (기계) | 장비, 도구, 기술 | 장비가 정상 작동하는가? |
| Material (재료) | 원자재, 부품, 공급 | 재료 품질에 문제는? |
| Method (방법) | 절차, 프로세스, 정책 | 표준 절차가 있는가? |
| Measurement (측정) | 데이터, 검사, 기준 | 측정 방법이 정확한가? |
| Mother Nature (환경) | 물리적 환경, 조건 | 환경 요인의 영향은? |

## 혁신/서비스업 적용 (5P + 1E)

```
                    ┌─ People (인력/이해관계자)
                    │
                    ├─ Process (업무 프로세스)
                    │
[혁신 실패/성공] ◄──┼─ Product (제품/서비스)
                    │
                    ├─ Policy (정책/전략)
                    │
                    ├─ Place (장소/채널)
                    │
                    └─ Environment (환경/문화)
```

## 혁신이론 적용 예시

### 혁신 실패 원인 분석
```
      People              Process            Technology
    (인력 역량)         (업무 방식)          (기술 수준)
        │                   │                   │
    ・리더십 부재       ・관료주의          ・기술 노후화
    ・저항 세력         ・의사결정 지연      ・통합 실패
    ・역량 부족         ・사일로 현상        ・보안 이슈
        │                   │                   │
        └───────────────────┼───────────────────┘
                            │
                    [혁신 실패] ◄────────────────────
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
    ・예산 제약         ・저항 문화          ・시장 변화
    ・시간 부족         ・위험 회피          ・규제 강화
    ・인프라 미비       ・비전 부재          ・경쟁 심화
        │                   │                   │
    Resources            Culture           Environment
    (자원)               (문화)             (외부환경)
```

## 작성 단계

### Step 1: 문제 정의
- 명확하고 구체적으로 문제 진술
- "왜?"로 시작하지 않기
- 측정 가능한 형태로 작성

### Step 2: 주요 원인 범주 선정
- 6M, 5P+1E, 또는 맞춤 범주 선택
- 보통 4-6개 범주 사용

### Step 3: 원인 브레인스토밍
- 각 범주별로 잠재 원인 나열
- "5 Whys"로 더 깊이 파고들기
- 모든 아이디어 환영

### Step 4: 분석 및 우선순위화
- 데이터로 원인 검증
- 근본 원인 식별
- 영향력/해결가능성 평가

### Step 5: 조치 계획
- 핵심 원인에 대한 해결책 수립
- 책임자 및 일정 지정

## 워크샵 진행 팁

### 준비물
- 큰 화이트보드 또는 전지
- 포스트잇 (색상별)
- 마커펜

### 진행 방법
1. 문제를 오른쪽에 적고 수평선 그리기
2. 주요 범주를 뼈대로 표시
3. 브레인스토밍으로 원인 나열 (포스트잇)
4. 각 원인에 "왜?" 질문 (세부 원인)
5. 투표로 핵심 원인 선정

## 활용 상황
- 품질 문제 해결
- 프로젝트 실패 분석
- 고객 불만 원인 파악
- 프로세스 개선
- 혁신 장애 요인 분석

## 장점
- 체계적 원인 분석
- 팀 협업 촉진
- 시각적 이해 용이
- 근본 원인에 집중
- 문서화 및 공유 편리

## 한계
- 원인 간 상호작용 표현 어려움
- 정량적 분석 부재
- 시간 소요
